# 涨了么小程序·产品架构优化方案
## 基于 Qiaomu AI PRD 的对标分析与迭代建议

**文档版本**：v1.0  
**分析时间**：2026-07-03  
**对标参考**：侵犯的 qiaomu 产品架构方法论 + 微信小程序最佳实践  

---

## 一、现状评估（Demo HTML → Mini Program 化）

### 1.1 当前架构的三个核心问题

| 问题 | 影响范围 | 优先级 |
|------|--------|--------|
| **单一信息源** | 只做聚合展示，无结构化业务飞轮 | P0 |
| **缺失B端数据链路** | 场馆主无法主动提供排期，冷启动难 | P0 |
| **社群与交易脱离** | 社群和预约是两个孤岛，未形成回环** | P1 |

### 1.2 对标 Qiaomu 的诊断逻辑

Qiaomu 的核心产品哲学：
- **一个核心链路**：用户 → 内容 → 行动 → 反馈 → 迭代
- **三层数据模型**：静态元数据 + 实时用户行为 + 众包反馈
- **双向激励**：创作者有动力投入内容，用户有动力参与反馈

**你的涨了么目前的问题**：
- ✅ 用户层完成了（筛选、预约、收藏）
- ❌ 场馆主层缺失（无数据输入、无激励、无反馈闭环）
- ❌ 社群层形同虚设（预约和社群无关联）

## 二、产品架构优化方案（分阶段）

### Phase 1.5（当前→2周）：补齐场馆主驱动链路

#### **1. 场馆主分层体验**

```
免费场馆主（Cold Start）
  ↓ 手机号注册（无需认证）
  ↓ 选择运动类型 + 排期模板
  ↓ 填写本周可用时段（15min）
  ↓ 获得专属管理码（二维码）
  ↓ [可选] 生成"空档推送链接"
```

**为什么分层**：
- Qiaomu 的核心是"让创作者愿意每天打开"
- 涨了么场馆主的痛点是"懒"，要把门槛降到2分钟
- 手机号注册 vs 微信认证：前者冷启动快，后者交易才需要

#### **2. 排期模板系统**（自动化排期输入）

```javascript
// 场馆主一次性设置"我的排期规律"
{
  venueName: "浦东网球中心",
  sportType: "网球",
  weeklyPattern: {
    周一至周五: { open: "07:00", close: "22:00", available_slots: [8, 9, 10, 11, ...] },
    周六周日: { open: "06:00", close: "23:00", available_slots: [...] },
  },
  blackout: ["2026-07-04", "2026-07-05"], // 休馆日
  // 自动生成每周排期，场馆主只需"确认"或"快速编辑"
}
```

**收益**：
- 场馆主周维护时间从30min → 5min
- 系统自动生成周排期，用户看到"当前可订"的实时感
- 这是韵动吧也无法做到的"边缘竞争力"

#### **3. 场馆主激励体系**

```
每日签到（2 credits）
  ↓
更新排期（5 credits）
  ↓
用户点"预约"成功（1 credit/次）
  ↓
积累 50 credits → 换"豪华商家展位"1周（首页置顶）
```

**对标 Qiaomu**：Qiaomu 的"创作者激励"驱动内容质量持续迭代。涨了么同样需要"场馆主激励"来驱动数据新鲜度。

---

### Phase 2（2-4周）：社群与预约闭环

#### **4. 社群与场馆的绑定模式**

当前痛点：用户预约后跳到各平台，与社群无关联。

**优化方案**：

```
用户点"预约" 
  ↓ [弹窗] "推荐加入该场馆球友群（企业微信）"
  ↓ 用户扫码入群（可选）
  ↓ 群内消息：
    - 场馆主推送排期更新
    - 用户可在群内聊天约球
    - 涨了么自动消息：空档提醒（企业微信 Webhook）
  ↓ 群活跃度 → 用户回访率 ↑
```

**为什么用企业微信**：
- 不需要微信小程序审批
- 100行代码的 Webhook，企业微信已提供 API
- 用户已在微信生态，转化成本低

#### **5. 用户众包反馈环**

```
用户预约 × 3次 后，弹窗：
  "帮助其他球友判断场地质量"
  [有空位✓] [质量好✓] [推荐指数]
  
积累50+用户反馈 → 生成"场馆评分卡"
  星级展示在场馆卡片
```

**对标 Qiaomu**：Qiaomu 的评论体系是质量反馈的来源。涨了么的众包反馈是"实时库存"的来源。

---

### Phase 3（4-8周）：数据与推荐引擎

#### **6. 分层数据模型（重建）**

从当前的"静态聚合"升级为"动态数据系统"：

| 数据层 | 当前 | 优化后 | 数据源 |
|--------|------|--------|--------|
| **第一层：静态元数据** | ✅ 有 | ✅ 完善 | 场馆主填报 + 人工审核 |
| **第二层：实时排期** | ❌ 无 | ✅ 新增 | 场馆主排期表 + 众包反馈 |
| **第三层：用户行为** | ✅ 有（HTML localStorage） | ✅ 上云（服务器落地） | 预约点击、收藏、搜索、筛选 |
| **第四层：社群热度** | ❌ 无 | ✅ 新增 | 群成员数、消息频率、预约转化 |

#### **7. 推荐算法三块（MVP）**

```
推荐模块 = 个性化 + 热度 + 业务

(1) 个性化推荐
    用户历史偏好 × 相似场馆
    → "你上次在浦东网球中心，类似的还有..."

(2) 热度排序（众包+社群）
    热度分 = (预约次数 × 0.5) + (群活跃度 × 0.3) + (用户评分 × 0.2)
    → 首页默认按热度排

(3) 智能筛选推荐
    用户打开筛选 5 秒后：
    → "我们为你预选了 3 个最匹配的场馆"
    → 一键应用
```

**为什么**：Qiaomu 的推荐 = 内容质量 + 用户参与度。涨了么的推荐 = 场馆热度 + 排期新鲜度 + 社群活跃度。

## 三、产品大图重构（IA改进）

### 当前 IA（问题在于：分散）
```
首页（聚合列表）
  ├─ 筛选
  ├─ 搜索
  ├─ 场馆卡片
  │  ├─ 收藏❤️
  │  └─ 预约按钮 → 跳转各平台
  ├─ 我的收藏
  ├─ 运营后台（管理员）
```

### 优化后 IA（核心：闭环）
```
首页 （推荐场馆 + 热度排序）
  ├─ [置顶] 我的关注场馆 + 群内消息提示
  ├─ 场馆卡片 v2
  │  ├─ 排期预览（本周可订时段）
  │  ├─ 社群信息（X人已加群）
  │  ├─ 用户评分 ⭐
  │  ├─ 收藏❤️
  │  └─ "加群+预约" 组合行为
  ├─ [Tab] 筛选 → 智能推荐辅助
  ├─ [Tab] 我的
  │  ├─ 我的收藏
  │  ├─ 我的群（快速跳转企业微信）
  │  ├─ 我的预约记录（聚合）
  │  └─ 个人资料 + 社群身份
  ├─ [Tab] 社群
  │  ├─ 发现球友（同场馆、同运动类型）
  │  ├─ 约球广场（球友发起的约球）
  │  └─ 赛事日历
  │
└─ 场馆详情页（新）
   ├─ 场馆基本信息 + 排期预览（本周）
   ├─ 用户评价聚合 + 众包反馈
   ├─ "加入球友群" CTA
   ├─ "立即预约"按钮组（多平台快捷）
   └─ 场馆主信息（可选留言）
```

## 四、技术架构升级建议

### 4.1 数据库 Schema 优化

```sql
-- 当前 HTML 版：localStorage 五个表
-- 升级后：Cloudflare D1 + 实时同步

CREATE TABLE venues (
  id INT PRIMARY KEY,
  name VARCHAR(255),
  location VARCHAR(255),
  ...
  week_pattern JSON,  -- 新增：排期模板
  updated_at TIMESTAMP  -- 新增：实时排期更新时间
);

CREATE TABLE venue_schedules (
  id INT,
  venue_id INT,
  date DATE,
  time_slot VARCHAR(10),  -- "08:00-09:00"
  capacity INT,
  available_slots INT,
  updated_at TIMESTAMP,
  FOREIGN KEY (venue_id) REFERENCES venues(id)
);

CREATE TABLE user_feedbacks (  -- 新增：众包反馈
  id INT PRIMARY KEY,
  user_id INT,
  venue_id INT,
  has_availability BOOLEAN,  -- "有/无空位"
  quality_score INT,  -- 1-5星
  created_at TIMESTAMP,
  FOREIGN KEY (venue_id) REFERENCES venues(id)
);

CREATE TABLE venue_groups (  -- 新增：场馆社群映射
  id INT PRIMARY KEY,
  venue_id INT,
  wechat_group_id VARCHAR(255),  -- 企业微信群ID
  member_count INT,
  activity_score FLOAT,
  FOREIGN KEY (venue_id) REFERENCES venues(id)
);
```

### 4.2 业务逻辑层（新增）

```
业务数据集成 (BFF - Backend for Frontend)

GET /api/v1/venues?sort=hot&limit=20
  |
  v
(1) 基础场馆数据
(2) 本周排期（场馆主最新输入）
(3) 用户众包反馈聚合 (avg_score, 反馈人数)
(4) 社群热度指标 (group_member_count, last_msg_time)
(5) 用户个性化标签 (已关注? 已加群?)
  |
  v
返回富文本卡片数据，前端直接渲染
```

## 六、与 Qiaomu 的架构对标总结

| 维度 | Qiaomu 做法 | 涨了么优化 | 核心启发 |
|------|--------|---------|--------|
| **用户链路** | 浏览 → 保存 → 分享 → 评论 → 反馈 | 筛选 → 加群 → 预约 → 反馈 → 推荐 | 链路必须闭环，每个点都有数据 |
| **创作激励** | 赞、评论数 + 排版工具 | 排期更新 + 积分 + 置顶展位 | 激励机制驱动内容新鲜度 |
| **推荐算法** | 内容热度 + 用户偏好 + 创作者质量 | 场馆热度 + 用户偏好 + 排期新鲜度 | 算法的三变量必须可量化 |
| **信息架构** | 发现 → 详情 → 评论社群 | 推荐 → 场馆详情 → 社群入口 | 从发现到社群的通路要顺畅 |
| **冷启动** | 编辑精选 + 推荐算法 | 场馆主激励 + 众包反馈 | 冷启动永远需要主动方参与 |
